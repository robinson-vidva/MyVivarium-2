# MyVivarium Chatbot — Response Formatting & Follow-Up Suggestions

This document describes two features layered on top of the existing
chatbot pipeline:

1. **Consistent response formatting** — driven entirely by system prompt
   rules so every reply has the same shape regardless of which tool was
   called.
2. **Follow-up suggestions** — up to two clickable chip buttons rendered
   below each AI reply, generated by the model with a rule-based fallback.

> The chatbot is currently in **testing phase**. The formatting rules may
> evolve as we observe real-world output; revise the rules in
> `includes/chatbot_helpers.php` rather than per-endpoint PHP.

---

## Response formatting rules

The block returned by `chatbot_response_formatting_rules_block()` is
prepended to every system prompt (in `chatbot_build_messages()` in
`ai_chat.php`). It is **not admin-editable** — admins can tweak the
free-form admin prompt in `ai_settings.system_prompt`, but the formatting
rules stay constant so output stays consistent.

The rules tell the model to:

- Lead with a one-sentence plain-text summary.
- Use a markdown table for 2+ records (max 10 rows, 3-5 useful columns,
  right-aligned numerics, with a trailing `... and N more` row when more
  exist).
- Use a bullet list of `Field name: value` for a single record, with a
  bold entity ID on its own line.
- Reply with a single `No results found for <criteria>.` line when zero
  rows return.
- For writes, lead with a one-sentence bolded-verb confirmation.
- Strip filler openers, emojis, error codes, and YYYY-MM-DD dates by
  default.

See the `chatbot_response_formatting_rules_block()` function for the
verbatim list of rules — those nine rules are the contract.

The chat panel renders markdown inline, so any markdown emitted by the
model lands in the UI as-is.

## Follow-up suggestions

### How they appear

After each AI reply, up to two pill-shaped chip buttons render
below the AI's message bubble. Clicking a chip inserts the chip's text
into the input box and submits it as a new user message.

While the next request is in flight the chips fade out and become
non-clickable.

### How they are generated

Two-step:

1. **AI generates them.** The system prompt (the
   `chatbot_follow_up_suggestions_block()` block) instructs the model to
   emit a special marker after its main reply:

   ```
   SUGGESTIONS::["question 1","question 2"]
   ```

   Rules: max 2, each under 50 chars, must be actionable (map to a real
   tool), no suggestions for pending-confirmation turns. If the model
   has nothing useful, it should emit `SUGGESTIONS::[]`.

2. **Backend parses and validates.** `chatbot_parse_suggestions()` in
   `includes/chatbot_helpers.php` extracts the marker line, validates
   the JSON is an array of strings, caps the array at 2 entries, and
   drops any item over 50 characters. The marker is stripped from the
   content before it reaches the user (a defensive sweep also strips any
   malformed `SUGGESTIONS::` line that survives parsing).

3. **Fallback.** When the AI returns no valid `SUGGESTIONS::` block,
   `chatbot_fallback_suggestions()` produces up to two plausible chips
   based on the last tool the AI called this turn. The map looks at the
   `operationId` and emits curated suggestions; if nothing matches it
   returns the default pair (`What can you do?`, `Show me a dashboard
   summary`).

### Fallback mapping

| Last tool called      | Fallback suggestions                                   |
| --------------------- | ------------------------------------------------------ |
| `listMice`            | Show details of the first mouse / Filter by alive status |
| `getMouse`            | Show this mouse's offspring / Show this mouse's cage history |
| `listHoldingCages`    | Show one of these cages in detail / Which cages have open tasks? |
| `getHoldingCage`      | List the mice in this cage / Add a maintenance note here |
| `listBreedingCages`   | Show lineage of the first one / Which have pups currently? |
| `getBreedingCage`     | Show this cage's lineage / List recent litters         |
| `listTasks`           | Show only tasks assigned to me / What's due today?     |
| `listReminders`       | Show this week only / What's overdue?                  |
| `listMyNotifications` | Show only unread / Mark all as read                    |
| `listMaintenanceNotes`| Show notes for one cage / Show only recent notes       |
| `getDashboardSummary` | What needs my attention today? / Show recent activity  |
| `listStrains`         | How many mice per strain?                              |
| `listIacuc`           | Which cages use this protocol?                         |
| _(no tool called)_    | What can you do? / Show me a dashboard summary         |

### Extending the fallback

1. Add a new key to the `$map` array in
   `chatbot_fallback_suggestions()` (`includes/chatbot_helpers.php`).
2. Each value is a list of 1-2 short strings, each ≤ 50 chars, written
   as a complete user message.
3. Add a test case to `tests/chatbot_suggestions_test.php`.

You do **not** need to update anything else — the parser, the widget,
and the persistence path are all schema-agnostic about tool names.

### Persistence

Suggestions persist in `ai_messages.suggestions_json` (a nullable TEXT
column). On conversation reload, `ai_chat_history.php` returns each
message's `suggestions` array; the widget renders chips for the most
recent assistant message only.

The migration is idempotent — `database/api_setup.php` adds the column
if missing.

### Prompt-injection note

The `SUGGESTIONS::` marker is parsed **only** out of the AI's own final
assistant content. Tool results are never run through
`chatbot_parse_suggestions()`. So a database record containing the text
`SUGGESTIONS::["evil"]` cannot inject suggestions into the chip bar.

## Per-response token usage display — testing phase

Below every assistant message bubble the widget renders a small muted
subtitle of the form:

```
245 prompt + 87 completion = 332 tokens
```

Numbers come from the LLM provider's `response.usage` field — the same
source that `ai_usage_log` rows are populated from, so the displayed
numbers exactly match what is logged.

A single user message can fan out into multiple LLM round-trips (initial
reply, tool-result follow-up, possibly several iterations up to
`MAX_ITERATIONS`). The token line shows **one total per user message,
not per internal round-trip** — `ai_chat.php` sums prompt and completion
across every round-trip in the turn and stores the result on the final
assistant message row (`ai_messages.tokens_json`).

The line is intentionally suppressed in three cases:

- **User messages** and **system events** (cancellations, errors) — no
  AI inference produced them, so there's no usage to show.
- **`pending_confirmation`** previews — the turn isn't final yet; the
  count is incomplete until the user confirms or cancels.
- **Missing or zero usage** — older messages (from before this feature)
  or any LLM response that omitted `usage` gracefully render no line at
  all, rather than `0 tokens` clutter.

Token data persists on the assistant message row, so reloading the
conversation via `ai_chat_history.php` brings the same line back.

This is a **testing-phase feature** designed to give us per-query cost
visibility while we tune providers and prompts. It can be hidden later
by gating `formatTokenLine()` in `includes/chatbot_widget.php` on a
user preference (e.g. a new `ai_settings` toggle).

## Token budget — testing phase

The combined hardcoded system prompt now contains three blocks (security
rules, response formatting rules, follow-up suggestions) plus the
admin-editable system prompt. Combined size is roughly **700–900
tokens**, and that is acceptable during testing.

No existing limits were lowered to "make room": `chatbot_context_message_limit()`,
the 4000-token conversation budget in `chatbot_build_messages`, and the
8000-token hard cap in `chatbot_fit_to_budget` are unchanged. If
estimation hits the ceiling during testing, the existing
`error_log("Chatbot turn estimated $est tokens (cap $hardCap, after
trim)")` line surfaces it — investigate and tune real signals rather
than silently dropping context.
